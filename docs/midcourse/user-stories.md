 

# User Stories - Due Dates & Tags

Feature set for the Task Tracker (Python/FastAPI backend, simple web frontend).
Grounded in the existing `Task` schema: `status` = `ToDo`/`InProgress`/`Done`,
`priority` = `Low`/`Medium`/`High`, string `id`, ISO 8601 timestamps.

**Target user:** a solo developer or small team managing work in a single shared task list.

**Out of scope:** authentication, user accounts, multi-tenancy / per-user task lists,
real-time updates, mobile app, notifications, production database or deployment.

## Feature 1 - Due Dates & Overdue Filter

| ID                                | Story                                                                                                                                                 | Acceptance Criteria                                                                                                                                                                                                                                                                                                                                                                                                                                        | Notes / Assumptions                                                                                                                                                                                        |
| --------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **DD-1**                    | As a**team member**, I want to set an optional due date on a task so that I know when it needs to be finished.                                  | • Create/Edit modal has a due-date field;`POST /tasks` and `PATCH /tasks/{id}` accept an optional `due_date` (ISO 8601 date, e.g. `2026-07-20`).<br /><br />• When set, the date is stored, returned by `GET /tasks`, and rendered on the card.<br /><br />• Leaving it blank saves the task with `due_date: null`; existing tasks without a due date are unaffected.                                                                       | New optional field on the`Task` model; nullable to keep it backward-compatible with tasks created before this feature.<br /><br /> Date-only (no time-of-day) to match a single-team, no-timezone scope. |
| **DD-2**                    | As a**team member**, I want to clear or change an existing due date so that I can adjust a deadline when plans shift.                           | • Editing a task lets me pick a different date (updates`due_date`) or empty the field (sets `due_date: null`), and Save persists the change via `PATCH /tasks/{id}`.<br /><br />• The card and any overdue flag re-render immediately to reflect the new value.<br /><br />• Clearing the date removes the overdue indicator if one was shown.                                                                                                    | Reuses the Edit modal flow (contract B8). No separate endpoint needed.                                                                                                                                     |
| **DD-3**                    | As a**team member**, I want tasks that are past their due date to be visually flagged so that I can spot what's late at a glance.               | • A task with`status` ≠ `Done` and `due_date` earlier than today is shown with an "Overdue" indicator (e.g. red badge/border).<br /><br />• A task due **today** is *not* overdue; a task with no due date is never overdue.<br /><br />• A `Done` task is never flagged overdue regardless of its due date.                                                                                                                           | "Overdue" is derived at read/render time from server date vs.`due_date` - not a stored field. <br /><br />Uses the server's local date as the reference (no per-user timezones, per scope).             |
| **DD-4**                    | As a**team member**, I want to filter the board to show only overdue tasks so that I can focus on catching up on late work.                     | • An "Overdue only" filter toggle is available; when on, only tasks meeting the DD-3 overdue rule are displayed.<br /><br />• The three columns and their`[data-count]` pills recount to reflect only the filtered tasks.<br /><br />• Turning the filter off restores all tasks; a column with zero matches still renders with its "No tasks" placeholder (preserves B4 behavior).                                                                   | Filtering is client-side over already-fetched tasks unless a server query param is preferred; either way it must not break existing column/count/drag behavior.                                            |
| **DD-5** *(failure case)* | As a**team member**, I want the app to reject an invalid due date so that I don't accidentally save a task with a broken or backdated deadline. | • Submitting a malformed date (e.g.`2026-13-40` or `"soon"`) returns `422`; the modal stays open and shows a field-level error on the due-date input (via `loc`, per B8).<br /><br />• Submitting a well-formed date **earlier than today** also returns `422` with a field-level error - backdating is not allowed; a date of **today or later** is accepted.<br /><br />• No task is created or updated when validation fails. | Reuses the existing 422 → field-slot error routing from the contract (B8).<br /><br /> Both malformed and past dates are blocked at write time.                                                           |

### Corrected AI Assumption - Feature 1

- **Assumption made:** the draft allowed backdating - a due date in the past was accepted at write time and merely flagged as overdue (original DD-5).
- **Correction:** backdating is **not** allowed. DD-5 now rejects any due date earlier than today with a `422`; only a date of today or later can be saved. Overdue flagging (DD-3) still applies to tasks whose due date has since passed after being validly set.

## Feature 2 - Tags / Labels

| ID                                | Story                                                                                                                                   | Acceptance Criteria                                                                                                                                                                                                                                                                                                                                                         | Notes / Assumptions                                                                                                                               |
| --------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| **TG-1**                    | As a**team member**, I want to add one or more tags to a task so that I can categorize related work (e.g. `bug`, `frontend`). | • Create/Edit modal accepts multiple tags;`POST`/`PATCH` accept a `tags` array of strings, returned by `GET /tasks` and shown as chips on the card.<br /><br />• Tags are trimmed and de-duplicated case-insensitively before saving.<br /><br />• A task with no tags saves successfully with `tags: []`.                                                     | Tags are free-text strings stored on the task (no separate tag entity, matching in-memory storage).                                               |
| **TG-2**                    | As a**team member**, I want to remove a tag from a task so that I can keep its labels accurate as the work changes.               | • Editing a task lets me delete an individual tag chip; Save persists the reduced list via`PATCH /tasks/{id}`.<br /><br />• Removing the last tag saves the task with `tags: []`.<br /><br />• The card re-renders to show only the remaining tags.                                                                                                                  | Reuses the Edit modal flow (contract B8).                                                                                                         |
| **TG-3**                    | As a**team member**, I want to filter the board by a tag so that I can see only the tasks for one category at a time.             | • Selecting a tag shows only tasks that carry it; the three columns and their`[data-count]` pills recount accordingly.<br /><br />• Clearing the tag filter restores all tasks; empty columns still render their "No tasks" placeholder (preserves B4).<br /><br />• The tag filter and the overdue filter (DD-4) can be combined, showing only tasks that match both. | Filtering is client-side over already-fetched tasks unless a server query param is preferred; must not break existing column/count/drag behavior. |
| **TG-4**                    | As a**team member**, I want to see the same tag styled consistently across all cards so that I can recognize a category by sight. | • A given tag string renders with the same chip color/style everywhere it appears.<br /><br />• Tag matching is case-insensitive so`Bug` and `bug` display as one category.<br /><br />• Chip styling does not alter card sort order (priority sort per B2 is unchanged).                                                                                            | Color can be derived deterministically from the tag text (e.g. a hash) - no stored color, keeping the in-memory model simple.                    |
| **TG-5** *(failure case)* | As a**team member**, I want the app to reject invalid tags so that cards stay readable and the data stays clean.                  | • An empty or whitespace-only tag is dropped rather than saved.<br /><br />• A tag exceeding the length limit (e.g. > 30 chars) or exceeding the per-task count limit (e.g. > 10 tags) returns`422` with a field-level error; the modal stays open and no task is saved.<br /><br />• Correcting the input and resubmitting saves successfully.                        | Limits keep cards legible on a shared board. Reuses the 422 → field-slot error routing from the contract (B8).                                   |

### Corrected AI Assumption - Feature 2

- **No incorrect assumptions by AI.** The tags/labels stories were reviewed and accepted as drafted; no correction was needed.

## Coverage

- **Happy paths:** setting/clearing due dates (DD-1, DD-2), overdue flagging (DD-3), overdue filtering (DD-4), tagging and untagging (TG-1, TG-2), tag filtering (TG-3), consistent tag styling (TG-4).
- **Failure cases:** DD-5 (malformed or backdated date) and TG-5 (invalid/oversized tags), plus negative checks embedded in DD-3 (done tasks not flagged), DD-4 / TG-3 (empty-column rendering), and TG-1 (blank/duplicate tag handling).
- **Scope check:** none of these introduce auth, accounts, per-user lists, real-time updates, notifications, or a persistent DB - all features operate on the shared in-memory task list.
